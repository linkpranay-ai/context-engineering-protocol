# Reproducibility Guide

Exact steps to rerun the two real measurement mechanisms named in
[`EVIDENCE-METHODOLOGY.md`](../EVIDENCE-METHODOLOGY.md) (§2 and §5) and get a
comparable result. This guide does not introduce new methodology — it is the
"how to actually run it" companion to that document's "what it means" and
[`evidence-record-template.md`](evidence-record-template.md)'s "how to record
it."

## Reproducing a token-efficiency benchmark (`graphify benchmark`)

Prerequisite: `graphify` installed (`uv tool install graphifyy` or `pipx
install graphifyy`), per
[`.github/skills/ult-codegraph/SKILL.md`](../.github/skills/ult-codegraph/SKILL.md).

1. From the root of the codebase you want to measure, build the graph:
   ```bash
   graphify update . --no-cluster
   ```
   Use a scoped subdirectory in place of `.` if the codebase mixes source
   with generated/vendored/other-language content — see `ult-codegraph`'s
   "Step 0 — scope the index" for how to decide.
2. Run the benchmark:
   ```bash
   graphify benchmark
   ```
   It reports corpus size (words/estimated naive tokens), graph size
   (nodes/edges), average query cost, and an overall reduction multiple,
   plus a per-question breakdown.
3. Record the result using
   [`evidence-record-template.md`](evidence-record-template.md)'s
   token-efficiency section: codebase identity (name, approximate size,
   language), the reported reduction figure, and the date run. This is a
   **measured** result per `EVIDENCE-METHODOLOGY.md` §5 — it came from
   running the tool, not from an estimate.

`graphify benchmark` is a one-off measurement, not part of the regular query
loop — no need to re-run it routinely on an unchanged codebase. Re-run it
after `graphify update .` has processed a substantial set of changes, if you
want an up-to-date figure.

## Reproducing a context-package usage report (`usage_report.py`)

Prerequisite: at least one context package generated by `ult-context-generate`
and consumed by a downstream skill that has written back addenda per
[`CONSUMING-CONTEXT-PACKAGE.md`](../.github/skills/ult-context-generate/CONSUMING-CONTEXT-PACKAGE.md)
step 9 (`contexts/<package-id>.yaml` plus one or more sibling
`contexts/<package-id>_<date>.addenda.yaml` files).

1. From the project root:
   ```bash
   python .github/skills/ult-context-generate/scripts/usage_report.py [--dir contexts/]
   ```
   `--dir` defaults to `contexts/`; pass it explicitly if your packages live
   elsewhere.
2. This writes `<dir>/USAGE_REPORT.md`, aggregating every `contexts/<id>.yaml`
   package it finds against the `cites.ctx_ids` unioned across that package's
   addenda files, and reporting which context items were never cited by any
   downstream consumer.
3. If no packages exist yet, the script still exits 0 and reports "No context
   packages found" — this is the normal pre-first-run state, not an error.
4. The report's "Token data" section is **measured**, not self-reported, when
   it has samples: it is built only from `tokens_used` values that a
   consuming skill's harness actually recorded (per
   `CONSUMING-CONTEXT-PACKAGE.md` step 9), never an estimate. Absent any
   samples, it says so plainly ("No measured runs yet") rather than
   substituting a number.

## What this guide does not cover

Fallback-relevance measurement (`EVIDENCE-METHODOLOGY.md` §1, surface 2) has
no defined procedure to reproduce yet — `EVIDENCE-METHODOLOGY.md` §4 names
this as an open gap, not something this guide can walk through until a
baseline is defined.
